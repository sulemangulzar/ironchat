The uncomfortable question

Your RAG works. How do you know?

"I tried a few queries and it looked fine" is a demo, not engineering. It can't tell you whether your last change helped or hurt.

You cannot improve what you don't measure. Every knob you touched yesterday — chunk size, k, the embedding model — you tuned by vibes.
The diagnosis

Bad answers are a retrieval problem.

Symptom

The answer is wrong, vague, or invented.

Instinct

"The model is bad." "Fix the prompt."

Reality

The model answered faithfully — from the wrong chunks.

Fix

Debug retrieval first. Generation second.

A perfect model with the wrong context still gives a wrong answer. No prompt rescues a chunk that was never retrieved.
The foundation

First, the golden set.

A list of queries, and for each one — which chunks actually answer it. That's the whole artifact.

golden = { "how much PTO do I get?" → [chunk 3, chunk 4] , … }
Size

15–25 queries is enough to start. Real phrasing, including the awkward ones.

Cost

Manual and boring. There is no shortcut, and that's the point.

Why it matters

Without it, every metric below is undefined and every "improvement" is an opinion.

This is the step everyone skips — and it's exactly why their RAG never gets better. No labels, no progress.
The vocabulary of quality

Four numbers

precision@k

Is it noisy?

Of what you retrieved, how much was relevant.

recall@k

Did you miss it?

Of everything relevant, how much you found.

MRR

How high?

Rank of the first relevant hit — 1/rank.

nDCG

The full picture

Rank-aware, and handles graded relevance.

They're deterministic. Same retriever, same labels, same number — every time. That's what turns "I think it's better" into recall@5: 0.62 → 0.89.
The asymmetry that decides everything

A miss is fatal. Noise is just expensive.

Low recall — unrecoverable

A chunk you never retrieved can never be answered from. The information isn't there. Nothing downstream can save you.

Low precision — recoverable

Junk rides along. It costs tokens and can distract — but a reranker can throw it away later.

k is a tug-of-war. Raise it → recall ↑, precision ↓, context floods. Lower it → precision ↑, recall ↓. So: protect recall first, fix precision downstream.

Why hybrid search exists

Meaning is lossy.

Embeddings compress text into meaning — so exact tokens blur. Ask for E-4021 and get a chunk that talks about errors, while the one holding the code sits at rank 9.

Dense · semantic

Wins: "lift" ≈ "elevator", intent, paraphrase.
Blind: codes, IDs, surnames, acronyms, API names.

Sparse · BM25 keyword

Wins: exact strings — E-4021, part numbers.
Blind: synonyms, meaning, anything reworded.

They fail on opposite queries. That's the whole argument for hybrid — not "more is better," but "their blind spots don't overlap."
Hybrid mechanics

Fuse by rank, not score.

A cosine of 0.82 and a BM25 of 14.3 aren't the same units, scale, or range. Adding them is nonsense; normalising them is fragile.

RRF(d)  =  Σlists   1 / ( k + ranklist(d) )     # k ≈ 60
Reciprocal Rank Fusion throws the scores away. Rank #1 in either list contributes a lot; rank #40 in both contributes almost nothing. Scale-free, no tuning, ~5 lines of Python.
Ask a better question

The user's question is a bad search query.

It's chatty, vague, full of pronouns — and it's a question, while your corpus is full of answers.

Rewrite

Conversational → retrieval-shaped.

Multi-query

N paraphrases → retrieve each → fuse with RRF.

HyDE

Write a hypothetical answer, embed that. Answers look like documents.

Conversational

"Does it roll over?" → resolve against history into a standalone query.

Your Day 5 chatbot has this bug right now. Multi-turn RAG breaks without it. The cost: every rewrite is an extra LLM call — latency and money on every query.
Re-ranking · cross-encoders

The shortlist, then the interview.

Bi-encoder — yesterday

Embeds query and document separately. Vectors precomputed → millisecond search over millions. But the two never meet.

Cross-encoder — the reranker

Reads query and document together, one pass. Models their interaction → far more accurate. Nothing precomputable, so it can't scan a corpus.

The discipline

One variable. Measure. Record.

config P@5 R@5 MRR nDCG@5
baseline · dense, k=5 0.40 0.55 0.61 0.58
+ hybrid (RRF) 0.48 0.79 0.74 0.71
+ query rewrite 0.52 0.85 0.78 0.76
+ rerank (50 → 5) 0.76 0.87 0.93 0.89
Keep the losing rows. A change that didn't help is evidence — deleting it is how a team re-tries the same bad idea in three months. (Numbers illustrative — yours will differ.)
The honest part

Measured ≠ solved.

Small samples lie

20 queries is noisy. A 0.02 move is noise. Don't celebrate the third decimal.

Retrieval ≠ answer

These say nothing about faithfulness. That's a separate, generation-side job.

Rerank isn't free

Latency + cost on every query. A product decision, not a free win.

Your labels are biased

You'll label the queries you already thought of. Get someone else to write them.

Your task — build it

Make yesterday's RAG measurably better

1 · Build a golden set

15–25 real queries over your corpus → the chunks that answer each. Include ones it can't answer.

2 · Measure the baseline

Hand-write precision@k, recall@k, MRR, nDCG@k. Score yesterday's dense retriever. No framework.

3 · Add hybrid, rewriting, rerank

BM25 + RRF · a query rewriter · a cross-encoder reranker (wide → narrow). One change at a time.

4 · Report the table

Before vs after, every row. Then decide: what ships, and what wasn't worth the latency?
