The problem

A model sits a closed-book exam.

It only knows its training data — frozen at a cutoff — and only sees what's in its context right now.

Blind

Can't answer from your private docs. It never saw your handbook, your tickets, your codebase.

Stale

The world moved on the day after training stopped. It doesn't know.

Confident anyway

It doesn't know what it doesn't know — so it fills the gap. That's hallucination.

The fix
Turn it into an open-book exam.

Retrieve the passages relevant to the question, drop them into the context, and let the model answer from them — with receipts.

Embeddings & vectors

Meaning becomes geometry.

An embedding model turns text into a vector — so passages that mean the same thing sit near each other in space.

cosine = angle between vectors same model for docs & queries "car" ≈ "automobile"
Retrieval · top-k

Pull the k nearest — nothing more.

The query becomes a point too. The k closest chunks get retrieved into the context.

k too small → misses the answer k too big → floods context, "lost in the middle" start at 3–5, then tune
Chunking

You can't embed a whole book at once.

Split documents into passages. Size and overlap decide whether retrieval finds a clean answer or a blurry one.

Chunk too big

Retrieval gets noisy; the real answer is diluted by paragraphs of unrelated text riding along.

Chunk too small

You sever a passage from the context it needs — the model gets a fragment that no longer means anything.

Overlap is cheap insurance: repeat a little text across boundaries so a sentence split down the middle still survives in one piece.
Vector databases

A Python list won't survive a million vectors.

Vector DBs do fast approximate nearest-neighbour search — the closest points, without scanning every one.

Qdrant · today

A real, purpose-built vector DB. Free tier, and runs in-memory in your notebook with zero setup.

Each point holds

vector + text + metadata. The metadata is what powers citations and filters.

Also out there

Pinecone, Weaviate, Chroma, Milvus, FAISS. Same core job, different trade-offs.

Hybrid search

Meaning is not enough.

Semantic search blurs exact tokens — a code like E-4021, a name, an acronym. Keyword search nails those but misses meaning.

Dense · semantic

Vectors. Understands "lift" ≈ "elevator." Blurry on exact strings.

Sparse · keyword

BM25. Nails "E-4021" and part numbers. Deaf to synonyms.

Hybrid · fused

Run both, merge with RRF — rewards what ranks high in either.

Next step up: re-ranking — a sharper model reorders the top candidates before they reach the prompt.
Citations & grounding

Answers with receipts.

You retrieved known sources — so the model can point back to them. Verifiable beats plausible.

Keep metadata

Source & page on every chunk, from the start.

Answer from context

Only from what was retrieved — nothing invented.

Cite it

[handbook · p.3] next to the claim.

Say "I don't know"

When it isn't in the context. Then validate.

The honest part

RAG is a system, not a switch.

Garbage in, garbage out

The answer can't beat the chunks retrieval found. No prompt rescues bad retrieval.

It can still ignore you

Context is provided, not obeyed. You verify.

Everything's a knob

Chunking, embeddings, k, search strategy — all need tuning on real questions.
