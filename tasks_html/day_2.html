Definition

What a tool actually is

A function you expose to the model, described so it can choose to call it. This is what "function calling" / "tool calling" means.

Reach

Get to data and APIs beyond the model's training.

Action

Actually do things — book, send, create, update.

Grounding

Return real facts instead of a confident guess.

≈1 min. Keep it short — they've met function calling in passing. The three gifts a tool gives a model: reach (data/APIs), action (side effects in the world), grounding (facts, which fights hallucination — callback to Day 4). Then pivot hard into the reframe on the next slide, which is the heart of the whole day.
The reframe that matters
A tool is an API you're writing — for an LLM to call.

The description isn't documentation. It's a prompt.

The model reads your tool's text to decide when to call it, whether to call it, and how to fill the arguments. Ambiguity there is a bug in your system.

≈1.5 min — land this hard. On Day 3 they learned a Pydantic schema is a contract between their code and a caller. Today the caller is an LLM: brilliant, literal, non-deterministic. So the description is not for other developers — it's read by the model at inference time as part of its prompt. A vague description is not a documentation smell; it's a functional bug that makes the model call the wrong tool, at the wrong time, with the wrong arguments. Everything in the next three slides follows from this one idea. Pause after "It's a prompt."
Design

Anatomy of a tool contract

Name

Unambiguous and action-oriented. search_orders, not handle.

Description

What it does, when to use it, when NOT to, and what it returns — for the model.

Parameters

Typed, described per-field, constrained (enums, required vs optional).

Returns

Structured and predictable — the model has to parse what you send back.

Errors

Recoverable messages the model can act on — never a raw stack trace.

Side effects

What it changes in the world — and the blast radius if it goes wrong.

≈2 min. Six parts. Spend most time on Description (next slide drills it) and Side effects (the two risk slides drill it). Quick hits on the rest: names are verbs; parameters carry their own descriptions and constraints so the model fills them correctly; returns must be structured because the model parses them; errors must be recoverable ("no order found for that email") so the model can retry or explain, not choke on a 500. Tell them: this is the same rigour as any service boundary — because it is one.
The make-or-break

Same function. Different words. Different behaviour.

Vague — a bug

name: "query_db_orders" description: "Execute order query"

The model guesses when to call it and what to pass. It fires at the wrong time and hallucinates arguments.

Clear — a contract

name: "search_customer_orders" description: "Search a customer's orders by date range, status, or total. Returns order details incl. items & shipping."

The model knows exactly when it applies, and what it gets back.

Show, don't just tell. Attaching 1–5 realistic example calls to a tool lifts parameter accuracy sharply (Anthropic reports 72% → 90%). Caveat: for some reasoning models, examples can hurt — so test it.
≈2 min. This is a real example from Anthropic's own tool-use guidance. Read both descriptions aloud in the model's "voice." The bad one gives the model nothing to decide with. The good one names the shape of the input and the return. The lesson: writing tool descriptions is prompt engineering — the exact skill they built in Week 1. Then the examples note: JSON schema says what's structurally valid; examples show what correct usage looks like (which optional params to include, conventions). This is the single highest-leverage fix for a flaky tool. They'll prove this in the notebook — deliberately break a description and watch the model misfire.
The details that bite

Parameters, returns & errors

Parameters

Give each one a type, a description, and constraints — enums, required vs optional. The model fills these; constraints keep it honest.

Returns

Structured and consistent. Whatever you send back becomes the model's next input — messy returns produce messy answers.

Errors

"no user found with that email" — not a 500 stack trace. Give the model something it can recover from.

Never trust the arguments. The model can hallucinate parameters or emit invalid JSON. Validate every argument in your code — with Pydantic — before you run anything. The schema is a hint to the model, not a guarantee to you.
≈1.5 min. Tie straight to Day 3: Pydantic is how you validate the arguments the model sends. Repeat OpenAI's and Anthropic's shared warning: the model does not always produce valid JSON and may invent parameters — so your code validates before executing. Errors: a recoverable message keeps the loop alive (the model apologises or retries); a raw exception dumped into context is noise the model can't act on. This slide is the bridge into safety — because "run anything" is exactly what you must be careful about.
Autonomy & risk · part 1

Which tools to ship — and which to leave out

Include

Tools that directly serve the app's job, and are low-risk or reversible:

  • Read-only: look up, search, calculate, fetch
  • Reversible writes: drafts, tags, rows you can delete
  • The narrowest capability that does the job

Leave out (or gate hard)

Out-of-scope functionality, or high-risk & irreversible actions:

  • Anything unrelated to the app's purpose
  • Delete permanently · send money · email outsiders
  • Run shell / arbitrary code · change permissions · deploy
Least privilege. Every tool you add is attack surface. A support bot does not need a "run SQL" tool. Give the model the smallest toolbox that gets the job done — nothing more.
≈2 min. This is the first of two risk slides and it answers the question directly: what belongs, what doesn't. Include = in-scope + (read-only or reversible). Exclude = out-of-scope (scope creep is pure attack surface for zero benefit) OR high-risk-and-irreversible. The out-of-scope point is underrated: interns love adding a "just in case" tool — every one widens what a jailbreak can reach. Next slide: for the risky-but-necessary tools, here's how you keep them.
Autonomy & risk · part 2

Tier every tool by what a jailbreak could do

TierWhat it can doHow you let it run
Tier 0 read-onlyLook up, search, calculate, fetch. No side effects.Autonomous. Still validate inputs. Safe to just run.
Tier 1 reversibleWrites you can undo — drafts, tags, rows it created.Autonomous, but logged. Make undo easy; add rate limits.
Tier 2 irreversibleDelete, send, pay, deploy, run code, touch others' data.Human approval required — or don't expose it at all.
The design test for every tool: "If a jailbroken model called this with the worst possible arguments — what's the damage, and can I undo it?" If the answer is "severe" and "no," it's Tier 2. Gate it behind a human, a confirmation step, a dry-run preview, or an allow-list — or cut it.
≈2.5 min — the payoff of the safety thread. Connect explicitly to Day 5: they hardened their bot against jailbreaks; now assume the jailbreak succeeds and ask what the tools let it reach. That's what tiering bounds. Walk the three tiers. Tier 0 runs freely. Tier 1 runs but you log and can undo. Tier 2 — irreversible, high-impact — never runs on the model's say-so alone: human-in-the-loop approval (their Week 4 LangGraph approval gate is exactly this), or a confirmation/preview step, or a tight allow-list (e.g., can only email internal addresses; can only delete rows it created). The one-line test on screen is the thing to remember. Least privilege + tiering means a successful jailbreak is contained, not catastrophic.
The pivot
One perfect tool is easy. Now imagine fifty.

Wall 1 — definitions

Every tool's schema sits in the prompt on every call, whether used or not.

Wall 2 — results

Every call and every result round-trips through the context window.

Both walls spend the one resource you can't grow mid-request. This is context engineering — tokens are money, latency, and accuracy.

≈1.5 min. The pivot from "one good tool" to "scale." Make the number real: five connected MCP servers (Jira, GitHub, Drive, Slack, a DB) can be 50,000+ tokens of tool definitions before the user even speaks. Two distinct problems, both draining the same finite context: (1) definitions loaded up front, (2) intermediate results piling up. Name it as context engineering and note the third cost people forget — a cluttered context also lowers accuracy, because the model has more to be distracted by. Tool search solves wall 1; programmatic calling solves wall 2.
Scaling the number of tools

Tool search

Don't load every tool up front. Mark most as discoverable; the model searches for the few it needs and loads only those.

OpenAI — built in

# Responses API · gpt-5.4+ tools=[ crm_namespace, {"type": "tool_search"} ] # defer_loading: true on the rest

Anthropic — built in

tools=[ {"type":"tool_search_tool_regex_20251119"}, # your tools: defer_loading: true ] # keep 3–5 hot tools loaded
Reported impact: tool definitions drop from tens of thousands of tokens to a ~500-token search tool; accuracy often rises (less clutter). Analogy: you don't memorise the library catalogue — you search for the book when you need it.
≈2 min. The mechanism: instead of all definitions in the prompt, you keep your 3–5 most-used tools loaded and mark the rest defer_loading: true. The model sees only a lightweight search tool plus names; when it needs a capability, it searches and the API injects just those definitions. Both platforms ship this now — OpenAI in the Responses API on gpt-5.4+ (group tools into namespaces for best results), Anthropic via the tool-search tool (regex or BM25). Bonus: because deferred tools aren't in the initial prompt, prompt caching stays intact. In the notebook they'll run OpenAI's native version and measure the input-token drop themselves.
Scaling the complexity of tool use

Programmatic tool calling

For multi-step, data-heavy work, the model writes code that orchestrates the tools. Only the final result returns to the context — the raw data never enters the conversation.

OpenAI — built in

Programmatic Tool Calling & the Code Interpreter tool run the model's code in a sandbox: {"type":"code_interpreter"}.

Anthropic — built in

{"type":"code_execution_20260120"}, # your tool: "allowed_callers":["code_execution..."]
Reported impact: ~38% fewer input tokens on a 75-tool benchmark — and tool results consumed in code don't count as tokens at all. Analogy: you don't read your assistant 500 numbers to total — you hand them a calculator. It powers spreadsheet and data agents.
≈2 min. The problem: "which of 20 employees exceeded their Q3 travel budget?" the naive way is 20 tool calls, each dumping 50–100 line items into context — thousands of wasted tokens. Programmatic calling: the model writes a Python loop that calls the tools, filters and aggregates in code, and returns only the names. Anthropic's version literally lets the model's sandboxed Python call your tools; OpenAI's Code Interpreter runs the model's Python in a hosted sandbox. Sequential single-call tasks don't benefit — this is for loops and data. And notice it also solves a privacy problem: sensitive rows can move between systems in code without ever entering the model's context. One caution to voice: executing model-written code demands a real sandbox — which is exactly why you use the managed feature rather than hand-rolling exec().
Decision guide

When to reach for each

TechniqueReach for it when…
Tool contractsAlways. The moment you give the model any tool at all. This is the foundation.
Tool searchYou have many tools — roughly 10+, or several connected services / MCP servers.
Programmatic tool callingA task is multi-step or chews through large data (loops, filters, aggregations).
Contracts are the floor; the other two are how the floor scales. You'll write contracts on day one. You'll reach for search and programmatic calling the first time a real toolbox gets big or a real task gets heavy.
≈1 min. Simple mental model. Contracts always. Tool search when the toolbox is big (schemas bloat the prompt). Programmatic calling when the work is big (data floods the context). Note the honest limit: a single, simple tool call needs neither of the scaling features — don't add machinery you don't need. Then move to the task.
