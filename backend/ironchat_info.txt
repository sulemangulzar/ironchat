# IronChat - Complete Project Documentation

## 1. Project Overview
IronChat is a modern, responsive, and elegant AI chatbot application designed to provide quick questions, helpful answers, and smooth conversational flows. It leverages the raw speed of Llama-3 through Groq's APIs to provide real-time streaming answers. IronChat acts as an advanced intelligent assistant that can answer general knowledge questions, perform live web searches for up-to-date information, and read uploaded documents (PDFs) to answer questions based on the user's private data.

## 2. Core Features
- **Blazing Fast AI Responses:** Powered by Llama-3 70b via Groq API for near-instant inference and low latency.
- **Premium UI/UX:** Built with Tailwind CSS v4, featuring a glassmorphic dashboard, responsive layout, dark/light mode toggles, and smooth micro-animations.
- **Secure Authentication:** Built-in user sign up, login, and session management using JWT access and refresh tokens, with passwords hashed via Argon2.
- **Optimized Backend:** A robust FastAPI backend integrated with PostgreSQL (via Supabase), using advanced async SQLModel techniques to minimize latency.
- **Streaming Messages:** Messages are streamed to the frontend via Server-Sent Events (SSE) for a typewriter-like feel, so users don't have to wait for the entire response to generate before reading it.
- **Web Search Integration:** Can intelligently decide when a user's prompt requires real-time information and perform live web searches using Tavily.
- **Retrieval-Augmented Generation (RAG):** Allows users to upload PDF documents, which the chatbot reads, processes, and uses as context to answer specific questions about the document's content.

## 3. Technology Stack

### Frontend (Client-Side)
- **Framework:** React 19 and Vite 8 for fast builds and hot module replacement.
- **Styling:** Tailwind CSS v4 for utility-first, highly responsive, and premium UI design.
- **Markdown Rendering:** `react-markdown` and `remark-gfm` are used to securely and beautifully render Markdown responses from the AI.
- **API Communication:** Custom Fetch wrappers are used to handle JWT authentication and Server-Sent Events (SSE) for real-time text streaming.

### Backend (Server-Side)
- **Language & Framework:** Python 3.13 and FastAPI, chosen for high performance and native async support.
- **Database (Relational):** PostgreSQL, hosted on Supabase.
- **ORM (Object-Relational Mapping):** SQLModel and SQLAlchemy 2.0 with `asyncpg` for high-speed asynchronous database operations.
- **Migrations:** Alembic is used to manage database schema changes.
- **Authentication:** PyJWT is used for JSON Web Tokens, and `pwdlib` with Argon2 is used for secure password hashing.

### AI, Machine Learning, and External APIs
- **Large Language Model (LLM):** Groq API running Llama-3 70b.
- **Web Search API:** Tavily (`tavily-python`), an AI-focused search engine API.
- **Vector Embeddings:** Voyage AI (`voyageai`), specifically the `voyage-3-lite` model, used to convert text into mathematical vectors.
- **Vector Database:** Qdrant (`qdrant-client`), used for fast semantic similarity search over document embeddings.
- **Document Parsing:** PyMuPDF (`fitz`), used to extract raw text accurately from uploaded PDF files.
- **File Storage:** Supabase Storage, used as a secure bucket to store the raw PDF files uploaded by users.

## 4. How the RAG (Document Chat) Pipeline Works
Retrieval-Augmented Generation (RAG) is the feature that allows IronChat to read and answer questions about PDFs.
1. **Upload & Storage:** The user uploads a PDF in the React frontend. The FastAPI backend receives it and stores the raw file in a Supabase Storage bucket.
2. **Text Extraction:** The backend uses PyMuPDF to extract text from the PDF, page by page.
3. **Chunking:** The extracted text is split into smaller, overlapping "chunks" (usually around 700 characters) to ensure the AI gets precise context without being overwhelmed.
4. **Embedding:** The text chunks are sent to the Voyage AI API, which converts the human text into vector embeddings (lists of numbers representing meaning).
5. **Vector Storage:** These vectors, along with metadata (like page numbers and chat IDs), are saved in the Qdrant Vector Database.
6. **Retrieval (Answering):** When the user asks a question, IronChat converts the question into a vector using Voyage AI, searches Qdrant for the closest matching chunks, and feeds those chunks into the Groq Llama-3 prompt so it can generate an accurate answer based on the document.

## 5. How the Web Search Pipeline Works
IronChat does not just rely on its pre-trained knowledge; it can search the live internet.
1. **Decision Phase:** When a user sends a message, a hidden system prompt asks the LLM: "Does this require a live web search? Answer yes or no."
2. **Search Phase:** If the LLM says yes, the backend uses the Tavily API to search the web for the query and retrieves the top results (titles, URLs, and text snippets).
3. **Augmentation:** The search results are injected into the context of the main conversation.
4. **Generation:** The Groq Llama-3 model reads the search results, writes a comprehensive answer, and includes inline citations (e.g., [1], [2]) linking back to the source URLs.

## 6. Security and Deployment
- **CORS & Origins:** The backend is strictly configured to only accept requests from the deployed frontend URL to prevent cross-site request forgery.
- **Deployment:** The frontend is deployed and hosted on Vercel. The backend is deployed and hosted on Render. 
- **Database Hosting:** The PostgreSQL relational database and the file storage buckets are hosted on Supabase.
